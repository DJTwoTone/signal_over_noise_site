# Signal over Noise — Pencil Update Handoff

## Purpose
This document captures the agreed update direction for the in-progress Signal over Noise website so it can be implemented in Pencil without restarting strategy work.

This is a refinement and launch-prep pass, not a rethink-from-scratch brief.

---

## Project snapshot

### What this site is
Signal over Noise is a premium, editorial-feeling, multi-page marketing site for a presentation coaching / presentation support business.

### Main goal
Generate leads and move visitors into the right next step.

### Main funnels

#### Main site funnel
Homepage or core page → Request a Free Presentation Diagnostic → intake / review → best next step → paid offer

#### Workshop QR funnel
`/scan` → toolkit opt-in → `/thanks` → Request a Free Presentation Diagnostic → intake / review → best next step → paid offer

### Locked homepage messaging
**Hero headline**  
Make your next presentation clearer, sharper, and more credible.

**Hero subhead**  
Signal over Noise helps professionals presenting in English and multilingual settings strengthen the message, refine the slides, rehearse the delivery, and prepare for high-stakes moments.

**Primary CTA**  
Request a Free Presentation Diagnostic

**Secondary CTA**  
Get the Signal over Noise Presenter Toolkit

### Audience framing
Use respectful audience language such as:
- professionals presenting in English and multilingual settings
- experts and teams presenting across international and mixed-language contexts

Do not use framing like “non-native speakers who need help.”

## Locked decisions from this review

### 1) Simplified nav
Use **4 links + 1 CTA**.

**Top nav**
- Services
- Process
- Proof
- Workshops
- Request a Free Presentation Diagnostic

**Rules**
- Remove **Home** from nav, logo links home
- Fold **Packages** into **Services** conceptually and in navigation
- Keep **Privacy** in footer only
- Keep `/scan` and `/thanks` out of the main nav

---

### 2) Homepage vs subpage content split
Use this rule:

- **Homepage = preview + persuasion**
- **Subpages = depth + detail**

#### Homepage should focus on
- core promise
- right next step
- fast credibility
- previews of available support

#### Subpages should expand with placeholder content for now
- Services: detailed support paths and offer mapping
- Process: diagnostic flow, recommendation logic, engagement steps
- Proof: expanded artifacts, founder credibility, later case studies/testimonials
- Workshops: workshop formats, use cases, outcomes, inquiry route

Do not mirror homepage copy word-for-word across subpages.

---

### 3) Contrast and readability pass
Run a full contrast/readability audit across the site.

#### Priority check areas
- nav text
- footer text
- eyebrow/label text
- small card text
- text on tinted backgrounds
- gold-accent text
- button text and hover states
- proof captions
- mobile readability

Treat this as launch-blocking QA, not optional polish.

---

### 4) Photo and icon expansion
Add more imagery selectively.

#### Best photo placements
1. Homepage hero / right-side visual module
2. Founder section
3. Workshops page
4. Proof page or proof section (supporting only, not primary)
5. Homepage audience/context support area
6. Services page (light use only)
7. `/scan` (optional, lower priority)

#### Must-source first
- Ben headshot
- Andrew headshot
- 1 strong editorial presentation photo
- 1 strong workshop/team-training photo
- custom icon set

#### Best icon placements
- homepage pathways
- process steps
- toolkit assets on `/scan`
- diagnostic review types
- possibly featured offers if needed

#### Style direction
- editorial
- real
- restrained
- intelligent
- not cheesy stock
- no mixed random icon styles

---

### 5) Hero proof treatment
Add proof to the hero, but keep it restrained.

#### Use
- a small micro-proof strip under the subhead or CTA row
- founder credibility or artifact-based proof before testimonial-style proof
- low-opacity diagnostic text as a near-transparent background layer

#### Background text treatment
- use real diagnostic language
- keep opacity very low
- treat it as texture, not content
- keep it mostly on the right side / behind the diagnostic module
- do not reduce readability of headline, body text, or buttons

Do not add a large testimonial block unless there is a real approved testimonial worth using.

---

### 6) Proof section rebuild
The homepage proof section should show evidence, not placeholder proof cards.

#### Homepage proof section = 3 blocks
1. **Sample diagnostic markup**
   - framed preview
   - 2–4 clear labeled callouts
   - short caption only

2. **Before / after slide comparison**
   - before left, after right
   - one short label about what improved

3. **Founder credibility strip**
   - compact trust support
   - not a full bio section

#### Proof page
Expand the deeper version later on `/proof`.

---

### 7) Founder section upgrade
The founder section should act as a credibility module, not a polite about-us section.

#### Direction
- Use a real Ben headshot
- Give Ben slightly more visual weight as founder
- Keep Andrew clearly distinct
- Keep both bios short, specific, and human
- Treat this as part of the proof system

#### Role distinction
**Ben**
- message shaping
- language
- deck refinement
- delivery confidence

**Andrew**
- message refinement
- storytelling clarity
- public-speaking support
- creative clarity

#### Suggested section heading
**Who’s behind Signal over Noise**

---

### 8) Diagnostic handoff clarity
Lock the diagnostic path sitewide.

#### Standard path
Request diagnostic → submit intake form → review → reply with the best next step

#### Rules
- intake comes first
- booking is follow-up when appropriate
- remove placeholder wording about turnaround/booking being undecided
- describe the diagnostic the same way everywhere

#### Working wording direction
Submit the request and we’ll review the presentation context, then reply with the best next step.

Optional version with turnaround:
Submit the request and we’ll review the presentation context, then reply within 1 business day with the best next step.

---

### 9) Homepage rhythm / card reduction
Keep the homepage structure, but reduce the repeated card-grid feel.

#### Section roles
- **Pathways** = quick self-identification preview
- **Process** = more linear, less card-like
- **Proof** = artifact-led evidence block
- **Featured offers** = short routing layer, not another services grid

#### Best homepage offer trio
- Presentation Diagnostic
- Presentation Sprint
- Presentation Coaching

Keep homepage sections from all feeling like the same repeated card treatment.

---

### 10) Services page absorbs Packages
Services should become the main detail page.

#### Services page should explain
- what kinds of support exist
- when each path fits
- what the visitor is buying into
- how the diagnostic connects to the right offer

#### Use the existing six pathways as structure
- Clarify the message
- Improve the deck
- Rehearse the delivery
- High-stakes presentation
- Build long-term skill
- Train the team

#### Then map into actual offers
- Free Presentation Diagnostic
- Presentation Sprint
- Presentation Coaching
- Message Refinement
- Slide Refinement
- Presentation Rehearsal
- Custom Workshops & Team Training

Keep the homepage as preview/routing. Move detail into Services.

---

### 11) `/scan` and `/thanks` as a deliberate mini-funnel
Treat these as a focused two-step workshop follow-up sequence.

#### `/scan` job
Get the toolkit opt-in.

#### `/thanks` job
Turn warm workshop traffic into diagnostic requests.

#### `/scan` should do
- toolkit value
- practical asset preview
- workshop continuity
- secondary mention of diagnostic

#### `/thanks` should do
- confirm toolkit
- explain next step clearly
- position the diagnostic as the natural continuation
- give one strong CTA

Do not let either page become a mini-homepage.

---

### 12) Sitewide CTA routing rules
Do not use the same two CTAs everywhere by default.

#### Page-by-page CTA map
**Home**
- Primary: Request a Free Presentation Diagnostic
- Secondary: Get the Signal over Noise Presenter Toolkit

**Services**
- Primary: Request a Free Presentation Diagnostic
- Secondary: optional workshop path where relevant

**Process**
- Primary: Request a Free Presentation Diagnostic
- Secondary: none or toolkit only if needed

**Proof**
- Primary: Request a Free Presentation Diagnostic
- Secondary: none

**Workshops**
- Primary: Ask About a Workshop
- Secondary: Request a Free Presentation Diagnostic

**`/scan`**
- Primary: Get the Toolkit
- Secondary: Request a Free Presentation Diagnostic

**`/thanks`**
- Primary: Request a Free Presentation Diagnostic
- Secondary: minimal or none

#### Additional rules
- nav CTA always goes to the same diagnostic destination
- footer CTA system should stay light
- remove placeholder inquiry language when routing is finalized

## Placeholder guidance
Use placeholders where structure needs to exist now but the final asset or copy is not ready.

### Good placeholder types
- expanded subpage copy blocks with clear internal notes
- image placeholders with exact sourcing direction
- icon placeholders by category
- workshop inquiry text placeholders if route is not fully set
- Andrew image placeholder if needed temporarily
- proof/support image placeholders

### Bad placeholder types
- lorem ipsum
- vague “coming soon” copy
- internal production notes showing in visible UI
- unresolved operational language presented as public copy
